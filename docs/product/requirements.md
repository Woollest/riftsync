## 1. 概要

RiftSync は、League of Legends のソロQ通常ランクにおいて、味方1人のピックをもとに、自分のロールで相性の良いチャンピオンを提案する初心者向けWebアプリである。

チャンピオンセレクト中に短時間で使えることを重視し、入力は最小限に抑える。おすすめチャンピオンを3体表示し、追加候補と非推奨候補も確認できるようにする。

現行版では、OP.GGのGlobal / Gold+ / Ranked Solo/Duoのロール別統計と個別シナジーページ、Riot Data Dragonのチャンピオン情報を利用する。直接相性データがある候補と推定補完の候補を区別し、信頼度と試合数を見ながら判断できることを重視する。

## 2. 目的

- 初心者が味方ピックとの相性を考慮して、自分のピックを選びやすくする
- メタ、勝率、コンボ相性を総合して、候補を3体に絞る
- 数値だけではなく、構成ベースの短い理由を表示する
- OP.GG直接データ、推定補完、信頼度、試合数を明示し、データの過信を防ぐ
- チャンピオンセレクト中でも迷わず使える、視認性の高いゲーム風UIを提供する
- URL共有、結果コピー、PWA対応により、友人にも使いやすい形で共有できるようにする

## 3. 想定ユーザー

- League of Legends 初心者
- ピック判断に慣れていないソロQプレイヤー
- 自分と友人が利用する範囲のユーザー

## 4. 対象範囲

### 4.1 対象

- League of Legends
- ソロQ
- 通常ランクのBAN/PICK
- Gold以上のデータ
- 地域指定なし
- チャンピオンセレクト中の利用
- GitHub Pagesで共有できるWebアプリ
- スマホのホーム画面に追加して使うPWA的な利用

### 4.2 初期版で対象外

- 敵ピックの考慮
- BAN状況の考慮
- 味方2体以上の組み合わせ考慮
- ARAM
- Clash
- カスタム
- スクリム
- 観戦、試合後分析
- Riotアカウント連携
- チャンピオンセレクト画面の自動読み取り
- 自動データ更新
- 管理者用データ更新画面
- 完全自動の定期データ更新

## 5. アプリ名

正式名称: RiftSync

名前の意図:

- Rift: LoLの世界観を想起しやすい
- Sync: 味方との連携、相性、構成のかみ合いを表す
- 短く、ゲーム風UIにも合わせやすい

## 6. 利用フロー

1. ユーザーが味方のロールを選択する
2. ユーザーが味方チャンピオンを検索またはアイコン一覧から選択する
3. ユーザーが自分のロールを選択する
4. アプリが自分のロールで使えるおすすめチャンピオンを3体表示する
5. アプリが追加候補を最大5体表示する
6. ユーザーが必要に応じて「非推奨候補を見る」を開く
7. アプリが非推奨チャンピオンを3体表示する
8. ユーザーが必要に応じて結果テキストまたは選択条件リンクをコピーする
9. スマホでは、必要に応じてホーム画面に追加してアプリのように起動する

## 7. 入力要件

### 7.1 味方のロール

必須入力。

選択肢:

- Top
- Jungle
- Mid
- ADC
- Support

### 7.2 味方チャンピオン

必須入力。

選択方法:

- 検索バー
- チャンピオンアイコン一覧
- 選択中の味方ロールにマッチするチャンピオンを上部に表示
- それ以外のチャンピオンを下部に表示

検索条件:

- 日本語名で検索可能
- 英語名で検索可能
- 部分一致検索

表示条件:

- チャンピオン基本データに存在する全チャンピオンを表示候補に含める
- 選択した味方ロールにマッチするチャンピオンを上部に表示する
- ロール一致判定は、ロール別統計データだけでなく、ポジション用チャンピオンカタログも使う
- それ以外のチャンピオンを下部に表示する
- 味方チャンピオンの選択一覧には「オフメタ」ラベルを表示しない

### 7.3 自分のロール

必須入力。

選択肢:

- Top
- Jungle
- Mid
- ADC
- Support

制約:

- UI上は、自分のロール選択で味方ロールと同じロールを選べないようにする

## 9. 表示ルール

### 9.1 総合評価

表示形式:

- 5つ星
- 半星あり

例:

- ★★★★★
- ★★★★☆
- ★★★★½

### 9.2 コンボ相性

表示形式:

- パーセント数値

例:

- コンボ相性 87%

### 9.3 勝率

表示形式:

- 数値をそのまま表示

例:

- 勝率 52.1%

### 9.4 メタ評価

表示形式:

- 炎マーク5段階
- 数字は併記しない

例:

- 🔥🔥🔥🔥

### 9.5 難易度

表示形式:

- 簡単
- 普通
- 難しい

データ元:

- Riot公式のチャンピオン難易度を利用する

変換基準:

- 1〜3: 簡単
- 4〜7: 普通
- 8〜10: 難しい

### 9.6 初心者におすすめラベル

表示条件:

- 難易度が「簡単」のチャンピオンに表示する

### 9.7 オフメタラベル

表示条件:

- 対象ロールにデータは存在するが、使用率が低い場合に表示する
- 初期基準は対象ロール内の使用率1.0%未満とし、実データを見て調整可能とする

### 9.8 データ少ラベル

表示条件:

- サンプル数が500試合未満の場合に表示する

除外条件ではない:

- データ少ラベルが付いていても、候補からは除外しない

### 9.9 推定相性ラベル

表示条件:

- CSVに直接の味方相性データがなく、チャンピオン性質ベースでコンボ相性を推定した場合に表示する

目的:

- 直接の統計相性と推定相性をユーザーが区別できるようにする

### 9.10 データ不足注意

表示条件:

- 直接相性データが3件未満の場合、推定相性を含むことを表示する
- 自分ロールの候補データが10体未満の場合、CSV拡充推奨を表示する

目的:

- データが少ない状態のおすすめを過信しないようにする

### 9.11 OP.GG直接 / 推定補完表示

表示条件:

- OP.GG個別シナジーページ由来の直接相性データがある候補には「OP.GG直接」と表示する
- 直接相性データがない候補には「推定補完」と表示する

目的:

- 統計上の直接相性と、チャンピオン性質からの推定をユーザーが区別できるようにする

### 9.12 信頼度表示

表示形式:

- 信頼度 高
- 信頼度 中
- 信頼度 低

初期基準:

- OP.GG直接データがあり、試合数が1000以上: 信頼度 高
- OP.GG直接データがあり、試合数が500以上1000未満: 信頼度 中
- OP.GG直接データがあり、試合数が500未満: 信頼度 低
- 直接相性データはないが、ロール統計の試合数が十分ある推定候補: 信頼度 中
- 補完データまたは直接相性データがない推定候補: 信頼度 低

### 9.13 追加候補表示

表示条件:

- おすすめ3体の推薦プールに続きがある場合、追加候補を最大5体表示する

目的:

- 上位3体が使えない場合の選択肢をユーザーに残す

## 10. スコア計算

### 10.1 総合スコア

総合スコアは以下の重みで算出する。

```text
総合スコア = コンボ相性 85% + 勝率評価 10% + メタ評価 5% - データ不足ペナルティ
```

重み:

- コンボ相性: 85%
- 勝率評価: 10%
- メタ評価: 5%

表示:

- おすすめカードには、総合スコアに対するコンボ相性、勝率評価、メタ評価の寄与を短く表示する
- 内訳は初心者が理解しやすいよう、細かい数式ではなく小さなバーと加点値で表示する

### 10.2 スコアの扱い

- 内部スコアは0から100で扱う
- コンボ相性は0から100のパーセントとして扱う
- 勝率は表示用の実勝率とは別に、同ロール内で正規化した勝率評価スコアへ変換する
- メタ評価は0から100の内部スコアとして扱い、表示時は炎マーク5段階に変換する
- 相性CSVが存在する組み合わせはCSVのコンボ相性を優先する
- 相性CSVが存在しない組み合わせは、味方チャンピオンと候補チャンピオンの性質から推定コンボ相性を算出する
- 推定コンボ相性では、エンゲージ、前衛、CC、ポーク、ピール、DPS、スケーリング、ダメージ種別などを使う
- コンボ相性の推定式には勝率やメタ評価を混ぜない
- サンプル数が500試合未満の場合はデータ不足ペナルティを強めに適用する
- サンプル数が500以上1000未満の場合は軽いデータ不足ペナルティを適用する

### 10.3 星評価への変換

総合スコアを5つ星、半星ありに変換する。

初期案:

```text
95-100 = ★★★★★
85-94  = ★★★★½
75-84  = ★★★★☆
65-74  = ★★★½☆
55-64  = ★★★☆☆
45-54  = ★★½☆☆
35-44  = ★★☆☆☆
25-34  = ★½☆☆☆
0-24   = ★☆☆☆☆
```

### 10.4 勝率評価スコアの正規化

勝率評価スコアは、同じ自分ロールの候補チャンピオン内で相対評価する。

初期案:

```text
勝率評価スコア = (対象勝率 - 同ロール最低勝率) / (同ロール最高勝率 - 同ロール最低勝率) * 100
```

補足:

- 表示上の勝率は実勝率をそのまま表示する
- 総合スコア計算には正規化後の勝率評価スコアを使う
- 最高勝率と最低勝率が同じ場合は、勝率評価スコアを50として扱う

## 11. おすすめ理由

### 11.1 方針

- 1行程度で表示する
- 数値ベースより構成ベースを重視する
- 初期版では固定テンプレート式にする
- テンプレートはロール別・組み合わせ別を含めて22種類用意する

### 11.2 理由テンプレート案

1. 味方のエンゲージに合わせて火力を出しやすい
2. 味方のCCに合わせてキルを狙いやすい
3. 前衛がいるため安全にダメージを出しやすい
4. 構成のダメージバランスを整えやすい
5. 味方のピールを受けながら継続火力を出しやすい
6. 味方の集団戦開始に合わせて範囲ダメージを出しやすい
7. 味方の序盤圧力に合わせてレーン主導権を取りやすい
8. 味方のスケーリングを支えやすい
9. 味方のロームやガンクに合わせて有利を広げやすい
10. 味方が作った隙にバーストで倒し切りやすい
11. 味方ADCに合わせて仕掛けの起点を作りやすい
12. 味方ADCの射程やハラスに合わせてレーンで圧力を出しやすい
13. 味方ADCの成長を守りながら安全に試合を進めやすい
14. 味方の前衛が時間を作るため安全に火力を出しやすい
15. 味方の拘束や集団戦開始に合わせて範囲火力を通しやすい
16. 味方JGと小規模戦を作りやすく川周辺で主導権を取りやすい
17. 味方の前衛に合わせて集団戦の範囲火力を通しやすい
18. 味方レーンのCCや圧力に合わせてガンクを刺しやすい
19. 味方の仕掛けに合わせて人数差やタワーダイブを作りやすい
20. 味方後衛を支える前線を作りやすい
21. 味方の集団戦圧力と分担してサイドで主導権を作りやすい
22. 味方と違うダメージ種別を足して対策されにくい構成にしやすい

### 11.3 非推奨理由テンプレート案

- 味方との勝率が低く、構成としてかみ合いにくい
- 味方の強みと噛み合うタイミングが少ない
- 集団戦で役割が重なりやすい
- ダメージや前衛のバランスが崩れやすい
- 味方の動きに合わせにくく、初心者には扱いづらい

## 12. UI要件

### 12.1 全体方針

- 日本語UI
- スマホ風の縦並びレイアウト
- 暗めでネオン感のあるゲームHUD風UI
- 配色は青・シアン系を中心にする
- アイコンを多用する
- チャンピオンセレクト中に素早く使える
- 操作数を少なくする
- データの信頼性がひと目で分かる
- スマホではホーム画面追加後もアプリらしく表示される

### 12.2 画面構成

1. ヘッダー
2. 味方ロール選択
3. 味方チャンピオン検索
4. チャンピオンアイコン一覧
5. 自分のロール選択
6. おすすめチャンピオン3体
7. 追加候補
8. 非推奨候補を開くボタン
9. 非推奨チャンピオン3体
10. 結果コピー / リンクコピー / フィードバック

### 12.3 レイアウト

- 基本は縦並び
- スマホ対応を前提にする
- PC横長表示では、左に入力欄、右にデータ情報・おすすめ結果を配置する
- チャンピオンアイコン一覧は小さいアイコンのグリッド表示にする
- 味方チャンピオン一覧では、ポジション用カタログに一致する候補を上段にまとめる
- スマホでは縦並びを維持し、ホーム画面追加時のsafe areaにも対応する

### 12.4 おすすめカード

カードに含める情報:

- チャンピオンアイコン
- 日本語名 / 英語名
- 総合評価
- 総合スコア内訳
- コンボ相性
- 勝率
- メタ評価
- 難易度
- 試合数
- OP.GG直接 / 推定補完
- 信頼度
- 各種ラベル
- おすすめ理由

### 12.5 PWA要件

- `site.webmanifest` を用意する
- スマホ用PNGアイコンを用意する
- Apple touch iconを用意する
- service workerを登録する
- ホーム画面追加後は standalone 表示を想定する
- PWAアイコンはスマホ側の角丸マスクで黒い縁やズレが出にくいよう、安全領域と視覚中心を調整する

### 12.6 読み込み・パフォーマンス要件

- チャンピオン画像は遅延読み込みする
- 重要なおすすめ1位の画像は優先表示し、2位以降や一覧画像は遅延読み込みする
- 検索入力は遅延反映し、入力中の描画負荷を抑える
- チャンピオン検索用インデックスを使い、検索時の文字列処理を抑える
- 大きな相性JSONは初期JSに含めず、別アセットとして起動後に読み込む
- 相性データ読み込み中はスケルトンカードを表示する
- Data Dragonと相性JSONはアプリ内でキャッシュし、重複リクエストを抑える

## 13. データ要件

### 13.1 データ参照元

参照元:

- OP.GG
- Riot Data Dragon

用途:

- OP.GG Champion Tier List: ロール別勝率、使用率、Tier、解析数
- OP.GG個別シナジーページ: 味方チャンピオンとの直接相性上位候補
- Riot Data Dragon: チャンピオン名、アイコン、公式難易度などの静的データ

注意:

- 外部サイトのデータ取得方法は利用規約や技術制約を確認する
- データ更新は管理者/開発者が手動でコマンド実行する
- 完全自動の定期更新は初期版では行わない

### 13.2 更新頻度

- 毎パッチ更新
- 自動更新は初期版では不要
- 管理者用画面は初期版では不要
- `pnpm update:opgg` でOP.GGのロール別統計とシナジーデータを取り込む
- 必要に応じて `data/manual/` のCSVを確認、調整する
- `scripts/import-csv-data.mjs` でCSVから `src/data/` のJSONを生成する

現行データ:

- ロール別統計: 259件
- 味方相性データ: 3108件
- 味方/味方ロール/自分ロールの直接相性グループ: 1036組
- 対象: OP.GG Global / Gold+ / Ranked Solo/Duo

### 13.7 データ作成・更新方針

- コンボ相性データはOP.GGの個別シナジーページから取り込む
- OP.GGのシナジー欄の上位3体を `pairSynergies.csv` に表示順で登録し、その3体をおすすめとして優先表示する
- ロール別勝率、使用率、Tier、解析数はOP.GG Champion Tier Listから取り込む
- チャンピオンアイコンと公式難易度はRiot Data Dragonを利用する
- CSV未登録の候補は、Riot Data Dragonのタグ/難易度/戦闘プロフィールとロール分類から補完候補として表示し、補完データであることを明示する
- 更新は管理者/開発者が手動コマンドで行い、定期自動実行は初期版では行わない
- OP.GGのHTMLやデータ構造変更で取り込みが失敗する可能性があるため、更新後は必ずデータ検証を行う
- 同じ味方ロール内でおすすめが固定化する場合は、該当する味方ロールから自分ロールへの直接相性データを優先して追加する
